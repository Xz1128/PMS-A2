Privacy & Security Analysis

Key privacy and security considerations for inventory data management.

1. Data Storage Security

This application stores inventory data in-memory within the browser. Data is not persisted to any server or database, which means it is lost when the browser tab is closed. While this eliminates server-side data breach risks, it also means there is no backup mechanism.

  • No server-side storage: Data never leaves the client machine, reducing exposure to network-based attacks.
  • Volatile memory: Data is held only in JavaScript runtime memory, not written to localStorage or cookies.
  • Recommendation: For production use, implement encrypted server-side storage with regular backups and access logging.

2. Input Validation & Sanitisation

Proper input validation is critical to preventing injection attacks and ensuring data integrity.

  • Type enforcement: Numeric fields (ID, quantity, price) use type="number" with min/max constraints to reject alphabetic input.
  • Required fields: All fields except Comment are validated as required before submission.
  • Enum validation: Category, Stock Status, and Popular Item are restricted to predefined values via dropdown menus.
  • ID uniqueness: Each Item ID is validated for uniqueness to prevent duplicate entries.
  • XSS prevention: Angular's built-in template binding automatically sanitises output, preventing cross-site scripting attacks.

3. Access Control

Currently, the application does not implement user authentication or role-based access control (RBAC). Any user with access to the application can add, modify, or delete inventory items.

  • No authentication: There is no login system. In a real deployment, implement JWT-based authentication.
  • No authorisation: All users have equal access. Consider implementing roles (e.g., Admin, Viewer, Editor).
  • Audit trail: Changes are not logged. Implement an audit log to track who modified what and when.
  • Session management: No session tokens are used. For production, implement secure session handling with CSRF protection.

4. Network Security

Since this is a client-side application with no backend API calls, network security concerns are minimal in the current state. However, if the application were to be extended:

  • HTTPS: All communications should be encrypted using TLS/SSL certificates.
  • API security: If a REST API is added, implement rate limiting, input validation on the server, and API key authentication.
  • CORS policy: Configure Cross-Origin Resource Sharing to restrict which domains can access the API.
  • Content Security Policy (CSP): Set HTTP headers to prevent loading of untrusted resources.

5. OWASP Top 10 Considerations

The following OWASP Top 10 security risks are relevant to inventory management systems:

  • A01 – Broken Access Control: Without authentication, access control is non-existent. Must be addressed before deployment.
  • A03 – Injection: Angular's template engine mitigates XSS, but server-side SQL injection must be guarded against if a database is used.
  • A05 – Security Misconfiguration: Default settings should be reviewed; error messages should not expose system internals.
  • A07 – Identification & Authentication Failures: No authentication is currently implemented. Use multi-factor authentication for sensitive operations.
  • A09 – Security Logging & Monitoring: Implement comprehensive logging for all CRUD operations and failed access attempts.

6. Data Privacy Compliance

If this system were to store personal data (e.g., customer names, supplier contact details), compliance with privacy regulations such as the Australian Privacy Act 1988 or GDPR would be required.

  • Data minimisation: Only collect data that is strictly necessary for the business purpose.
  • Right to access & deletion: Provide mechanisms for data subjects to request access to or deletion of their personal data.
  • Data retention policy: Define how long inventory and supplier data is retained and when it should be purged.
  • Consent: Obtain explicit consent before collecting or processing personal information.
